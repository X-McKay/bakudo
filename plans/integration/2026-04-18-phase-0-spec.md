---

## Goal

Make the bakudo↔abox dogfood path work end-to-end on a virtiofsd-capable host, without workarounds like `--abox-bin "$(which abox)"` or scripted approval bypasses. Give the operator enough diagnostic signal to distinguish host-setup failure from task failure even within Phase 0 (via the F-P preflight slice). Clear the still-valid subset of the 2026-04-18 test report. Make `abox --capabilities` succeed against an unmodified abox with an observable, parse-validated acceptance.

## Non-goals

Deliberately **out of scope**; covered by later phases:

- Worker extraction into a standalone npm package — **Phase 2**.
- Full cross-boundary deep-doctor (boots sentinel VM, schema-validates probe, checks clock skew, etc.) — **Phase 3**. Phase 0's F-P is strictly a minimal preflight slice that detects obvious host-setup problems *before* dispatch.
- Shared contract schema with generated types — **Phase 1**. Phase 0 uses a short written contract note (see appendix) instead.
- Full typed error-class taxonomy keyed to attribution buckets — **Phase 3**.
- Any fix not explicitly named in this spec.

If during implementation something new surfaces that feels like a Phase 0 blocker, add it to this spec and re-request approval; otherwise defer.

## Scope — 12 items

### F-00. Implement `abox --capabilities` as a top-level flag

**Repo:** abox
**Files:** `crates/abox-cli/src/main.rs` (add flag, change command parse, bypass like `init`/`doctor`), `crates/abox-cli/tests/capabilities_test.rs` (new)

Add `--capabilities` as a **top-level flag** on the root `abox` command (not a subcommand). bakudo probes it that way: `bakudo/src/host/workerCapabilities.ts:121` calls `execFn(input.bin, ["--capabilities"], ...)`.

**Clap wiring (implementation-critical):** Today `Cli.command: Commands` is mandatory (`abox/crates/abox-cli/src/main.rs:31-32`). For `abox --capabilities` to parse at all, the struct must either (a) make command optional — `command: Option<Commands>` — and add `#[arg(long)] capabilities: bool` as a sibling field, or (b) introduce a new `Commands::Capabilities` variant combined with a clap argument group that accepts `--capabilities` in root position. (a) is lower-risk: it keeps all existing subcommands unchanged and only adds a new "no subcommand + --capabilities flag" path. The main-fn dispatch then checks `cli.capabilities` first and short-circuits before reaching the subcommand match.

The handler **must bypass config and orchestrator loading** the way `init` and `doctor` already do (`abox/crates/abox-cli/src/main.rs:83-102`). Otherwise a misconfigured host aborts before the JSON is printed — exactly the case bakudo needs to diagnose.

Envelope shape — matches `bakudo/src/host/workerCapabilities.ts:85-107` exactly; three arrays, no ghost fields:

```json
{
  "protocolVersions": [1, 3],
  "taskKinds": ["assistant_job", "explicit_command", "verification_check"],
  "executionEngines": ["agent_cli", "shell"]
}
```

Array contents mirror bakudo's existing constants in `bakudo/src/protocol.ts`:
- `protocolVersions` = `BAKUDO_HOST_PROTOCOL_VERSIONS` = `[1, 3]`
- `taskKinds` = `BAKUDO_HOST_TASK_KINDS` = `["assistant_job", "explicit_command", "verification_check"]`
- `executionEngines` = `BAKUDO_HOST_EXECUTION_ENGINES` = `["agent_cli", "shell"]`

**Deliberate deferrals:** `aboxVersion` and `workerInRootfsVersion` are **not** in the Phase 0 envelope. bakudo's current validator ignores extra fields, so adding them now would be invisible and brittle. Phase 2 adds `workerInRootfsVersion` concurrent with bakudo-worker extraction; Phase 1 canonizes either field as part of the schema.

**Acceptance.** Integration test in bakudo spawns the live abox, captures stdout, and checks `source === "probe"`. `validateCapabilitiesJson` is currently private (`bakudo/src/host/workerCapabilities.ts:85-107`); Phase 0 uses the public `probeWorkerCapabilities` helper from the same module (`:115-162`) which already wraps validation and returns the `ProbeOutcome` carrying `source`. The test asserts `outcome.capabilities.source === "probe"` (not `"fallback_host_default"`) against a live abox. No private export is needed. **Not** "doctor no longer says `v1 (assumed)`" — that acceptance is weaker than the existing `doctorAboxProbe.ts:58-68` which passes on any non-empty stdout. Tightening `doctorAboxProbe` to parse-validate is listed as a **follow-up candidate** (see hand-off section) rather than a Phase 0 blocker.

**Why.** bakudo already calls this flag. Its fallback is optimistic-by-design (`hostDefaultFallbackCapabilities()` asserts the host's own capability set). Implementing `--capabilities` is ~30 lines in abox and lights up the Phase 6 W3 safety net.

### F-03. Resume reads from `attemptSpec`, not legacy `request`

**Repo:** bakudo
**Files:** `src/host/sessionLifecycle.ts:140-189`, plus any code that composes a retry `WorkerTaskSpec` from the stored attempt

Current behavior (schema mismatch, not a state-loss bug): `executeAttempt.ts:133-138` persists the attempt with `attemptSpec: spec`. `resumeSession` at `sessionLifecycle.ts:144-145` reads `attempt.request` and throws `no resumable attempt found for session <id>` when it's undefined — which is always, because dispatch writes `attemptSpec` rather than `request`.

Change: migrate `resumeSession` to source the retry from `attempt.attemptSpec`. Remove the `attempt.request === undefined` guard and replace with `attempt.attemptSpec === undefined`. Preserve the existing success/blocked/policy_denied short-circuits (`sessionLifecycle.ts:150-157`).

**Conversion path (implementation-critical):** `AttemptSpec` and the legacy `WorkerTaskSpec` are different shapes (`bakudo/src/attemptProtocol.ts:61` vs `bakudo/src/workerRuntime.ts:20`). Current resume uses `executeTask` (the legacy worker path). Phase 0 picks one of two concrete conversion routes:

1. **Preferred:** convert via the existing adapter helper at `bakudo/src/aboxTaskRunner.ts:220`, which already knows how to compile an `AttemptSpec` into a dispatchable `WorkerTaskSpec`. Resume reads `attempt.attemptSpec`, runs it through that helper, and hands the result to `executeTask` unchanged.
2. **Alternative:** reroute resume to `runAttempt` (the attempt-native path) entirely, skipping `executeTask`. Higher leverage but larger diff.

Phase 0 picks route (1) to keep diff bounded. Route (2) is a Phase 1 candidate once the schema codifies `AttemptSpec` as the canonical dispatch input.

**Legacy session compatibility:** bakudo's session migration (`bakudo/src/sessionMigration.ts:111`) already preserves `request` for v1 sessions during upgrade. Resume reads `attempt.attemptSpec ?? attempt.request` with the `request` branch routed through the same conversion helper in route (1). The v1 branch is a single function call, not a parallel code path.

**Acceptance.** Regression test: dispatch a failing build against a mock that returns a retryable failure; confirm `attempt.attemptSpec` is persisted; run `resume`; assert a second attempt is dispatched with `turnId === "turn-1"` and `attemptId` incremented.

### F-04. Preserve PATH through env filter for host-side spawn

**Repo:** bakudo
**Files:** `src/host/envPolicy.ts`, `src/aboxAdapter.ts`

Current behavior: `envPolicy.ts:40-43` defaults to an empty allowlist; `filterEnv` at `:64-77` drops every name not explicitly allowlisted — including PATH. When `aboxBin` is a bare name ("abox", the default), Node's `spawn` needs PATH for binary resolution. The result is `spawn abox ENOENT`.

Change: in `ABoxAdapter.runInStreamLive()`, when the spawn target has no path separator (i.e., the binary is an unqualified name), inject `PATH: process.env.PATH` into the child env unconditionally — regardless of the user's env-allowlist. This PATH is used only for binary resolution on the host side; the guest VM inherits its own environment from abox's rootfs init, so nothing leaks into the sandbox.

Document in the envPolicy module header: "PATH is always preserved on the host spawn when `aboxBin` is an unqualified name, because it is required for binary resolution and never reaches the guest."

**Acceptance.** Regression test: construct an `EnvPolicy` with `allowlist: []`; call `ABoxAdapter.runInStreamLive()` with `aboxBin: "abox"`; inspect the `spawn` env argument and assert PATH is present. Integration test: run `bakudo build` without `--abox-bin` on a shell where `which abox` resolves, assert no `ENOENT`.

### F-05. `-p --output-format=json` never prompts plaintext — without coupling to authority

**Repo:** bakudo
**Files:** the `-p` entry point (`src/host/oneShotRun.ts` and `src/host/sessionController.ts:224-231`), `src/host/copilotFlagParser.ts`

Current behavior: `bakudo -p "hello" --output-format=json` prints a plaintext approval prompt in the middle of the JSONL stream, then blocks on stdin. Any programmatic consumer breaks.

**Critical design constraint from external review:** `-p` + `--output-format=json` must **not** imply `autoApprove=true`. Authority (approve-all-tools) and output format (JSON vs text) are orthogonal semantic axes. The earlier revision conflated them by wiring `--output-format=json` to also grant execution authority — that is a bad coupling.

Change: two sub-fixes, both localized to output semantics only.

1. **Never prompt in JSON output mode.** When `--output-format=json` is set, the approval code path must not write to stdout and must not block on stdin. This applies to the top-level prompt `promptForApproval` in `src/host/oneShotRun.ts:61-69` and to any session-controller approval hook (`sessionController.ts:224-231`).

2. **Emit a typed error when approval is required.** If the session would have prompted, emit a single JSONL event:

   ```json
   {"ok":false,"kind":"error","error":{"code":"approval_required","message":"Approval is required but stdin is --output-format=json","details":{"taskId":"...","mode":"build"}}}
   ```

   and exit with the existing `EXIT_CODES.BLOCKED` (2) from `bakudo/src/host/errors.ts:27` — **do not invent a new exit code**. The user can then retry with an explicit `--allow-all-tools` or `--yes` if they want to grant authority separately.

**New string code (add to `BakudoErrorCode`):** `"approval_required"` is not currently in the `BakudoErrorCode` union at `bakudo/src/host/errors.ts:41-50`. Phase 0 adds it to that type. This is distinct from the existing `"approval_denied"`, which means "the user was asked and declined"; `"approval_required"` means "we could not ask because output is machine-readable." Different recovery advice, different downstream handling. The addition is recorded in Appendix A.3.

**Acceptance.** Regression test: run `bakudo -p "hello" --output-format=json` with a denying policy (mocked); stdout contains only JSONL (every line starts with `{`); no TTY-blocking call; exit code = `EXIT_CODES.BLOCKED`. Test asserting that **setting `--output-format=json` alone** does NOT change `autoApprove` in the resolved options.

### F-P. Minimal host-preflight slice

**Repo:** bakudo
**Files:** extend `src/host/doctorAboxProbe.ts` with host checks (or a sibling module `src/host/hostPreflight.ts`)

**Goal.** Distinguish host-setup failure from task failure during Phase 0. Report clearly when the host cannot boot an abox sandbox.

**Checks in scope for Phase 0 (exactly two):**

1. **virtiofsd file capabilities.** Shell out to `getcap <configured-virtiofsd-path>`; if output is empty or lacks `cap_sys_admin+ep` (or the equivalent user-mapping setup), report a single actionable error:
   ```
   [ERR] host-virtiofsd-caps: virtiofsd at /path/to/virtiofsd lacks required capabilities.
         Fix: sudo setcap 'cap_sys_admin+ep' /path/to/virtiofsd
   ```
2. **/dev/kvm access.** Stat `/dev/kvm`; if missing or not user-readable/writable, report:
   ```
   [ERR] host-kvm-access: /dev/kvm not accessible for uid=<n>.
         Fix: sudo usermod -aG kvm $USER (log out and back in)
   ```

**Rootfs presence check deferred.** A "does the rootfs exist at the path abox will use" check would require either a stable `abox run --check-only` probe (does not exist in `abox/crates/abox-cli/src/main.rs:35-` today) or brittle text-scraping of `abox doctor`. Neither is minimal. Rootfs presence/freshness moves to Phase 3, where abox's `--capabilities` envelope (post-Phase-2) will include `workerInRootfsVersion` and the deep doctor will parse-validate it.

**Checks deliberately out of scope for Phase 0:**

- Full deep-doctor VM boot — Phase 3.
- Clock skew — Phase 3.
- Worker version match — needs Phase 2 to exist.
- Rootfs presence/freshness — Phase 3 (see above).
- Schema-validating `--capabilities` output — a Phase 3 enhancement to `doctorAboxProbe`.

**Acceptance.** On a host with virtiofsd lacking caps, `bakudo doctor` exits nonzero with the `host-virtiofsd-caps` error surfaced before any task dispatch is attempted. On a fully-provisioned host, both checks pass.

## Appendix: minimal contract note (lock-down for F-00 and exit codes)

Per the external critique, Phase 0 touches two load-bearing surfaces that Phase 1 will canonize. To avoid shipping F-00 without written-down shape, and to avoid Phase 0 fixes re-inventing exit codes, this appendix documents the bounded contract Phase 0 commits to:

### A.1 `abox --capabilities` envelope (Phase 0 version)

```jsonc
{
  "protocolVersions": [1, 3],       // numeric array, length >= 1
  "taskKinds": [                    // string array, length >= 1
    "assistant_job",
    "explicit_command",
    "verification_check"
  ],
  "executionEngines": [             // string array, length >= 1
    "agent_cli",
    "shell"
  ]
  // NO OTHER FIELDS IN PHASE 0.
  // Phase 2 adds workerInRootfsVersion (string, semver).
  // Phase 1 canonizes the full schema.
}
```

Validator: `bakudo/src/host/workerCapabilities.ts:85-107` (`validateCapabilitiesJson`). Any additional field is silently ignored by that validator today, so Phase 0 declines to emit any.

### A.1.1 Root-flag parse and bypass

abox's `Cli` struct is modified so `command: Option<Commands>` (currently mandatory). A sibling `#[arg(long)] capabilities: bool` accepts `--capabilities` in root position. `main` checks `cli.capabilities` before the subcommand match and, when true, runs the `capabilities` handler — which **must bypass config/orchestrator load** exactly as `init` and `doctor` do today (`abox/crates/abox-cli/src/main.rs:83-102`). The handler prints the JSON envelope to stdout and returns `Ok(())` without touching `~/.abox/config.toml`, policy engines, or runtime dirs. Phase 0 does not modify any existing subcommand's parsing; the only structural change is `command` becoming `Option`.

### A.2 Exit-code taxonomy (Phase 0 references, does not redefine)

The existing source of truth is `bakudo/src/host/errors.ts:24-32`:

```ts
EXIT_CODES = {
  SUCCESS: 0,
  FAILURE: 1,
  BLOCKED: 2,
  POLICY_DENIED: 3,
  PROTOCOL_MISMATCH: 4,
  SESSION_CORRUPTION: 5,
  SIGINT: 130,
}
```

Phase 0 fixes MUST use these constants. Specifically:
- F-05 uses `EXIT_CODES.BLOCKED` (2) for the `approval_required` error, not a new code.
- F-06 uses `EXIT_CODES.FAILURE` (1) for the unknown-key error.
- F-14's error path uses `EXIT_CODES.FAILURE` (1).

Phase 1 will canonize this table in the shared contract schema. Phase 0 does not modify the numeric values or the name set.

### A.3 Error-code string additions (Phase 0 adds one)

The existing source of truth is the `BakudoErrorCode` union at `bakudo/src/host/errors.ts:41-50`. Phase 0 adds exactly one new string:

- **`"approval_required"`** — emitted by F-05 when `--output-format=json` blocks the prompt surface and the session would have asked the user for approval. Distinct from the existing `"approval_denied"` (user asked, said no). Recovery hint in the rendered error: "re-run with `--allow-all-tools` or `--yes` to grant authority non-interactively."

No other additions in Phase 0. Phase 1 canonizes the full set.

## Testing strategy

### Unit tests (fast, in `tests/regression/`)

One test per fix ID, named `tests/regression/F-NN-<slug>.test.ts`. Each test constructs the minimum session/mock required and asserts the expected behavior.

### Integration tests (`node --test`, no VM required)

- `tests/integration/abox-capabilities.test.ts` — invokes the public `probeWorkerCapabilities` helper (`bakudo/src/host/workerCapabilities.ts:115-162`) against the live abox binary and asserts `outcome.capabilities.source === "probe"` (not `"fallback_host_default"`). This is the observable acceptance for F-00 and requires no private export.
- `tests/integration/spawn-abox-path.test.ts` — exercises F-04's PATH preservation with a constructed empty-allowlist `EnvPolicy`.
- `tests/integration/oneshot-json-no-prompt.test.ts` — drives `oneShotRun` with `--output-format=json` and a mocked approval-required path; asserts stdout is pure JSONL and exit is `EXIT_CODES.BLOCKED`. Covers F-05.

### E2E test (KVM-gated, nightly `tier-vm`)

Extend `just integration-test` with:
1. `git init` a scratch repo.
2. `bakudo build "add a top-of-file comment" --repo <scratch> --yes`.
3. Assert exit 0.
4. Assert README diff.
5. Assert `bakudo status` reports `mode=build`.

This requires virtiofsd-capable CI. Stays in `tier-vm` (nightly), not `tier-ci` (every PR). If CI cannot provision virtiofsd, the integration test runs locally pre-merge and F-P's preflight exercises the failure path instead.

## Acceptance criteria

On a virtiofsd-capable host:

1. `bakudo --version` → `bakudo <version>`, exit 0.
2. `bakudo -V` → same.
3. `bakudo resume <failed-retryable-id>` → dispatches a second attempt successfully.
4. `bakudo build` without `--abox-bin` on a shell where `which abox` resolves → no `ENOENT`.
5. `bakudo -p "..." --output-format=json` → only JSONL on stdout; no TTY-blocking prompts.
6. `bakudo -p "..." --output-format=json` with a denying mock → single `approval_required` JSONL error, exit `EXIT_CODES.BLOCKED`.
7. `bakudo doctor --explain-config nonsense.key` → `harness_error: unknown config key: nonsense.key`, exit 1.
8. `bakudo sandbox <id>` → under ~20 lines output.
9. `bakudo logs <id>` → no literal `undefined` in rendered fields *(if still broken at HEAD; else drop item)*.
10. `bakudo build` → progress line(s) during the pre-first-event window *(if still silent at HEAD; else drop item)*.

## Risks

### F-00 contract lock-in
abox growing a `--capabilities` top-level flag with a specific JSON shape is a **contract statement** the rest of the plan depends on. A future abox that renames fields or changes array contents breaks older bakudo. Mitigation: the Appendix A.1 note makes the shape explicit; Phase 1's schema extraction enforces it via CI.

### F-05 approval semantics
Emitting `approval_required` as a JSONL error when `--output-format=json` is set changes the failure shape a scripted caller sees. Callers that piped a `y` on stdin to auto-accept will now get an error envelope instead. Mitigation: this is the correct behavior — silent auto-acceptance is exactly the authority-coupling the reviewer flagged. Callers that actually want auto-approve pass `--yes` or `--allow-all-tools` explicitly.

### F-03 migration compatibility
Sessions persisted with `request` set (older bakudo versions) must still be resumable. Mitigation: read `attempt.attemptSpec ?? attempt.request`; schedule `request` removal for Phase 1 once the schema has a deprecation path.

### F-04 env-filter escape
Injecting PATH through the allowlist could be perceived as a policy bypass. Mitigation: PATH is injected only on the host spawn (required for Node's binary resolution) and is not forwarded to the guest VM — the guest gets its own environment from rootfs init. Document clearly; add a unit test asserting PATH is NOT present in the `WorkerTaskSpec` env field.

### F-P surface area creep
A "minimal" preflight has a natural tendency to grow into a full doctor suite. Mitigation: the spec explicitly enumerates the two Phase 0 checks (virtiofsd caps + `/dev/kvm`) and marks everything else (rootfs presence/freshness, clock skew, schema validation, sentinel VM boot) as Phase 3. If a case seems essential during implementation, add it to this spec and re-request approval.

### Stale-spec risk itself
The original revision of this spec carried five items already fixed at HEAD. Mitigation: the baseline-check table above was built from an HEAD verification pass, and F-08/F-13 are marked "provisional — re-verify." Implementers must sanity-check against HEAD one more time before starting.
